If AVAL/AVALC is imputed or if the entire record is imputed we will populate the DTYPE variable. So, a non-null value in DTYPE indicates that the record is imputed.
PARAMTYP variable is used for indicating whether the parameter on a particular observation is derived parameter or a collected parameter. PARAMTYP variable will populated 'DERIVED' for derived parameters and will be left null for collected parameters.
DTYPE variable is used for indicating whether a particular record is a collected record or a derived record. DTYPE variable will be populated with appropriate keyword (like LOCF, AVERAGE, MINIMUM, MAXIMUM etc) indicating the derivation algorithm used for creating the derived record and it will be left null for collected records.
BASETYPE variable is used to indicate the baseline definition used for populating BASE variable on a particular record. It is used only when there is more than one definition of baseline within a dataset or parameter. Examples of different definitions include: 1) Last known value prior to treatment start 2) Minimum value on screening period 3) Maximum value on screening period 4) Average result of last two collections prior to treatment start 5) Average of all results available prior to treatment start
The term 'Last Observed Value' is generally used to indicate the last known result while the subject is on study treatment. In ADaM implementation, we can use two methods to indicate such record. 1) Create a flag variable and populate it as 'Y' on the last observed value. 2) We can create an additional record as a copy of the record corresponding to the last observed value. Second approach is preferable when last observed value is presented as a conceptual timepoint in a by-visit analysis.
In clinical trial data, we often encounter partial dates (one or more components of a date — year, month, day — are not known). For analysis purposes like calculating approximate durations, we fill the missing components with prespecified values. This process is called date imputation. In ADaM, if an imputed date value is stored in any analysis date variable (ADT, ASTDT, AENDT etc) we use date imputation flag variable -DTF to indicate the level of imputation performed. -DTF will have a value of Y when Year is imputed, M when month is imputed, D when day is imputed.
Similarly, for time imputations, we use -TMF variable to indicate the level of imputation. -TMF will have a value of 'H' if hour is imputed, 'M' when minutes are imputed, 'S' when seconds are imputed.
When there exists multiple records at the lowest level of time point precision, we generally create a new record to hold the average of the result and use it in analysis. If the result is of qualitative nature on those records we may choose either the best case or worst case based on the analysis requirement by creating a flag variable to choose the appropriate record.
Based on analysis requirements, we may impute death date if it is partial if it is prespecified in the statistical analysis plan.

SMQ stands for Standardized MedDRA Queries. As part of evaluation of a drug's safety, reviewers may be interested in checking for the presence of a group of adverse events that are related by medical concept (e.g., hepatotoxicity, anaphylaxis). SMQ variables in ADAE (e.g., SMQ01NAM, SMQ01CD, SMQ01SC, SMQ01SCN) allow a single AE record to be flagged as belonging to one or more predefined MedDRA query groupings, enabling efficient safety signal detection across the entire adverse event dataset without needing separate subsets per query.

  • ADaM IG v1.0 was the first published version establishing the foundational ADaM structures: ADSL, BDS, and OCCDS classes.
  • ADaM IG v1.1 introduced clarifications on traceability, additional guidance on DTYPE, and refinements to BDS variable definitions.
  • ADaM IG v1.3 is the most widely used version at FDA submissions today. It significantly expanded guidance on occurrence datasets (OCCDS), time-to-event datasets (ADTTE), and period/subperiod structures for cross-over studies.
  • ADaMIG v1.4 and beyond continue to refine guidance, add new dataset classes, and align with CDISC 360 traceability goals.
  • In practice, always confirm the required version with the sponsor and regulatory agency before starting dataset development, as FDA and PMDA may specify different version requirements.

Traceability in ADaM means that every analysis result can be traced back to the source data — from the TFL output, through the analysis dataset, all the way to the raw SDTM or collected data.

  • Why it matters: Regulators (FDA, EMA) require that analysis results be independently reproducible. Traceability enables reviewers to verify that results are correctly derived from source data.
  • How it is achieved: ADaM datasets include source variable references (e.g., LBTESTCD maps to PARAMCD), SRCDOM/SRCVAR/SRCSEQ variables where applicable, and define.xml documents that formally describe every variable and its lineage.
  • Bidirectional traceability: You should be able to go forward (raw data → ADaM → TFL) and backward (TFL → ADaM → raw data) at the record level.
  • Audit trail: The combination of ADaM datasets and their define.xml together constitute the full traceability package submitted to regulators.

In short: SDTM preserves collected data integrity; ADaM transforms that data into a form ready for statistical analysis.

The define.xml for ADaM is a machine-readable metadata document (XML format, conforming to the CDISC Define-XML standard) submitted to regulatory agencies alongside the ADaM datasets.

  • Dataset-level metadata: Dataset name, label, class, structure, unique keys, purpose, and location on submission media.
  • Variable-level metadata: Variable name, label, data type, length, codelist reference, controlled terminology, origin (collected, derived, assigned), and computational method for derived variables.
  • Value-level metadata: For variables like AVAL or DTYPE that have different derivations depending on PARAMCD, value-level metadata documents each derivation separately.
  • Analysis results metadata (ARM): Optionally links each dataset/variable to the specific analysis result in the SAP/TFL, providing end-to-end traceability.
  • Codelists: All controlled terminology values used in the datasets are enumerated.

The define.xml, together with the analysis datasets, is a mandatory part of every regulatory submission package.

  • ADSL (Subject Level): One record per subject. Used for demographics, population flags, treatment assignment, and study completion information. Every study must have an ADSL.
  • BDS (Basic Data Structure): One or more records per subject per analysis timepoint per parameter. Used for continuous measurements — labs, vitals, efficacy endpoints. Examples: ADLB, ADVS, ADEFF.
  • OCCDS (Occurrence Data Structure): One or more records per subject per occurrence of an event. Used for discrete events — adverse events (ADAE), medications (ADCM), medical history (ADMH).
  • ADTTE (Time-to-Event): A specialized BDS class. One record per subject per event parameter. Used for survival analysis — time to first AE, progression-free survival, overall survival.
  • Other/Custom structures: For datasets that do not fit the above (e.g., ADPC for pharmacokinetics), the structure is documented in the define.xml and the analysis plan.

Analysis Results Metadata (ARM) is an optional but highly recommended extension to the ADaM define.xml that provides a formal, machine-readable link between analysis datasets and the specific results reported in TFL outputs.

  • What it documents: For each reported result, ARM captures the dataset used, the WHERE clause applied (e.g., SAFFL='Y'), the analysis variable (AVAL), the analysis reason, the statistical method, and the resulting display reference (table/figure number).
  • Why it matters: ARM enables full end-to-end traceability from a table cell back to the analysis dataset records and ultimately to raw SDTM data. FDA reviewers can use ARM to independently reproduce results.
  • Regulatory expectation: FDA's Technical Conformance Guide encourages submission of ARM. It is increasingly expected for NDA/BLA submissions.
  • Practical benefit: During audits or deficiency responses, ARM significantly reduces the time needed to explain how a specific result was generated.
  • Flag variables (e.g., ABLFL, ANL01FL, TRTEMFL) are Yes/null indicators that mark records meeting a specific condition. They are used to select records for a particular analysis without creating separate datasets. A flag of 'Y' means the record qualifies; null means it does not.
  • Criterion variables (CRIT1, CRIT1FL, CRIT2, CRIT2FL, ...) describe a clinical threshold or responder definition applied to AVAL. CRIT1 holds the text description of the criterion (e.g., 'AVAL >= 30') and CRIT1FL holds 'Y' or 'N' for each record depending on whether that record's AVAL meets the criterion.
  • Key difference: Flags select records for inclusion in an analysis population or analysis window; criterion variables evaluate the magnitude or clinical significance of the result on each individual record.
  • Both types enable the downstream SAS programmer to generate a table using a simple WHERE clause without re-deriving analysis logic.

  • STUDYID: The unique identifier for the clinical study. It is the same value used across all SDTM and ADaM datasets for a given trial. It allows multi-study pooled analyses to distinguish records by study.
  • USUBJID: Unique Subject Identifier — a globally unique subject identifier formed by concatenating STUDYID and SUBJID (e.g., 'STUDY001-001-0042'). It is the primary key for linking records within and across datasets at the subject level.
  • SUBJID: Subject Identifier within the study — a shorter identifier (e.g., '001-0042') that may be more readable but is only unique within a study.
  • In ADaM, USUBJID is the universal join key between ADSL and all other analysis datasets. Every subject-level record in every ADaM dataset must have a corresponding record in ADSL with the same USUBJID.

In SDTM, the SUPPQUAL (Supplemental Qualifiers) dataset stores additional variables that do not fit within the standard domain structure. Each record in SUPPQUAL has RDOMAIN (parent domain), USUBJID, IDVAR/IDVARVAL (key linking back to parent record), QNAM (variable name), and QVAL (value).

  • In ADaM creation: Supplemental qualifier variables are typically transposed (PROC TRANSPOSE or a data step with conditional logic) from the long SUPPQUAL structure back into a wide format so they can be merged onto the parent domain by USUBJID + the ID variable.
  • Once merged onto the SDTM domain, the variable is available as a regular column and can be used in ADaM derivations like any other source variable.
  • Traceability: In define.xml, the origin of such variables should reference the SUPPQUAL dataset and the QNAM value to maintain full traceability.
  • Common examples: SUPPAE.AELLT (low-level term), SUPPDM.COMPLT16 (completion flag), SUPPEX.EXROUTE if not in main EX.
  • ANL01FL is a generic analysis flag that marks the records included in the primary analysis for a given parameter. It answers the question: 'Which record at each visit should be used for the primary analysis?'
  • Additional flags (ANL02FL, ANL03FL, ...) are created when a secondary or sensitivity analysis requires a different record selection rule for the same parameter. For example, ANL01FL may select the last non-missing value per visit while ANL02FL selects all observed values.
  • When to create a new flag: Create a new ANLxxFL whenever two analyses of the same parameter use different record-selection criteria. This avoids duplicating the dataset or re-deriving the selection in the TFL program.
  • All flag definitions must be documented in the ADaM specs and define.xml with a clear computational method so that any programmer can reproduce the flagging logic independently.
