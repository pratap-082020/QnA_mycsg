PROC SQL is SAS's implementation of ANSI SQL. It provides a query language interface to SAS datasets and supports most SQL features.

When PROC SQL is better than data step:
  • Joins — especially many-to-many, complex conditions, or outer joins. No need to pre-sort.
  • Aggregation — GROUP BY, HAVING, SUM, COUNT, MEAN in a single step.
  • Subqueries — easily express "find subjects who..." logic.
  • Dictionary tables — query metadata about datasets, variables, libraries.
  • Single-step read + summarize — often cleaner than PROC MEANS + MERGE.
When data step is better:
  • Complex row-by-row logic, retain, first./last., arrays.
  • Writing multiple output datasets.
  • Need full control over observation flow.
Basic syntax:
proc sql;
  create table result as
  select usubjid, age, sex
  from adsl
  where safflag='Y'
  order by usubjid;
quit;

Key syntax notes:
  • Ends with QUIT, not RUN.
  • Use SELECT * (with SAS dataset features), but explicit columns are cleaner.
  • Case-insensitive.
PROC SQL supports standard SQL joins:

INNER JOIN (or just JOIN):
  • Returns only matching rows from both tables.
  • SELECT a.*, b.value FROM t1 a INNER JOIN t2 b ON a.id = b.id;
LEFT JOIN:
  • Returns all rows from the left table, matched rows from the right. Unmatched right rows become NULL.
  • SELECT a.*, b.lab FROM adsl a LEFT JOIN adlb b ON a.usubjid = b.usubjid;
  • Clinical use: Keep all subjects even if they have no lab data.
RIGHT JOIN: Mirror of LEFT JOIN. Returns all rows from right, matched from left.

FULL JOIN: Returns all rows from both tables, NULLs where no match.

Implicit JOIN (Cartesian with WHERE):
FROM t1, t2 WHERE t1.id = t2.id — older syntax, still common in SAS. Functionally similar to INNER JOIN but less clear.

Example — Left join to add baseline value:
proc sql;
  create table adlb as
  select a.*, b.aval as base
  from lb a
  left join baseline b
  on a.usubjid = b.usubjid and a.paramcd = b.paramcd;
quit;

Advantages over data step MERGE:
  • No need to sort both datasets first.
  • Handles many-to-many relationships cleanly.
  • Join conditions can be complex (non-equi joins, multiple conditions).
GROUP BY aggregates rows by one or more columns. Aggregate functions (SUM, COUNT, MEAN, MIN, MAX) compute values across each group.

HAVING filters groups AFTER aggregation (vs WHERE which filters rows BEFORE aggregation).

Example — Count AEs per subject:
proc sql;
  select usubjid, count(*) as n_aes
  from ae
  group by usubjid;
quit;

With HAVING — subjects with more than 5 AEs:
proc sql;
  select usubjid, count(*) as n_aes
  from ae
  group by usubjid
  having count(*) > 5;
quit;

WHERE vs HAVING:
  • WHERE — filters rows before grouping. Cannot reference aggregate functions.
  • HAVING — filters after grouping. Can reference aggregate functions.
Example combining both:
select trt, count(*) as n_sae
from ae
where aeser='Y'
group by trt
having count(*) >= 10;

Common aggregate functions:
  • COUNT(*), COUNT(DISTINCT var)
  • SUM, MEAN, MIN, MAX, MEDIAN, STD
  • N (non-missing count), NMISS (missing count)
Clinical use: Efficacy summary stats per treatment, AE counts per subject, dose accountability.
A subquery is a query nested inside another query. PROC SQL supports subqueries in WHERE, SELECT, and FROM clauses.

IN subquery — set membership:
select * from adsl
where usubjid in (select usubjid from ae where aeser='Y');
Returns ADSL records for subjects who had at least one SAE.

NOT IN subquery:
select * from adsl
where usubjid not in (select usubjid from ae);
Subjects with no AEs.

EXISTS subquery:
select * from adsl a
where exists (select 1 from ae b where b.usubjid = a.usubjid and b.aeser='Y');
Similar to IN, but uses correlation between outer and inner query. Often faster for large data.

Subquery in SELECT:
select usubjid,
      (select count(*) from ae b where b.usubjid = a.usubjid) as n_aes
from adsl a;
Computes AE count per subject inline.

Subquery in FROM (derived table):
select a.*, b.n_aes
from adsl a
left join (select usubjid, count(*) as n_aes from ae group by usubjid) b
on a.usubjid = b.usubjid;

Clinical use: Flagging subjects meeting criteria defined in another dataset, excluding subjects based on events, computing per-subject metrics inline.
CASE WHEN provides SQL's if-then-else equivalent — used to derive conditional values in SELECT statements.

Simple CASE:
select usubjid,
  case sex
    when 'M' then 'Male'
    when 'F' then 'Female'
    else 'Unknown'
  end as sex_label
from adsl;

Searched CASE (with full conditions):
select usubjid,
  case
    when age < 18 then 'Pediatric'
    when age < 65 then 'Adult'
    when age >= 65 then 'Elderly'
    else 'Unknown'
  end as agegr
from adsl;

CASE with aggregation:
select trt,
  sum(case when aeser='Y' then 1 else 0 end) as n_sae,
  count(*) as n_ae
from ae
group by trt;
Counts SAEs and total AEs per treatment group.

CALCULATED keyword — reference derived columns in same SELECT:
select age,
  case when age >= 65 then 1 else 0 end as elderly,
  case when calculated elderly = 1 then 'E' else 'A' end as grp
from adsl;

Clinical use: Deriving analysis flags, creating subgroup variables, conditional summary statistics.
Dictionary tables are SAS's built-in metadata catalogs — they expose information about libraries, tables, columns, options, titles, and more. Accessible only via PROC SQL.

Common dictionary tables:
  • DICTIONARY.TABLES — datasets in all libraries.
  • DICTIONARY.COLUMNS — all variables in all datasets.
  • DICTIONARY.MEMBERS — datasets and catalogs.
  • DICTIONARY.OPTIONS — current SAS system options.
  • DICTIONARY.TITLES — current titles and footnotes.
  • DICTIONARY.FORMATS — available formats.
  • DICTIONARY.VIEWS — defined views.
Example — list all datasets in WORK:
proc sql;
  select memname, nobs
  from dictionary.tables
  where libname='WORK';
quit;

Example — find all datasets containing variable 'USUBJID':
proc sql;
  select libname, memname
  from dictionary.columns
  where upcase(name)='USUBJID';
quit;

In data step — use SASHELP views:
  • SASHELP.VTABLE (= DICTIONARY.TABLES)
  • SASHELP.VCOLUMN (= DICTIONARY.COLUMNS)
Clinical use: Automating metadata checks, building dynamic pipelines (loop through all ADaM datasets), QC scripts to verify variable presence.
Use the INTO clause in a SELECT statement to assign query results to macro variables.

Single value:
proc sql noprint;
  select count(*) into :n_subj
  from adsl where safflag='Y';
quit;

%put Number of safety subjects: &n_subj;

Multiple macro variables (one per row):
proc sql noprint;
  select count(*) into :n_mild, :n_mod, :n_sev
  from ae
  where aesev in ('MILD', 'MODERATE', 'SEVERE')
  group by aesev;
quit;

Concatenate values into a list:
proc sql noprint;
  select distinct quote(trim(trtp)) into :trt_list separated by ','
  from adsl;
quit;

%put Treatments: &trt_list;
Output: Treatments: "Drug A","Drug B","Placebo"

Indexed macro variables (one per row, auto-numbered):
proc sql noprint;
  select distinct trtp into :trt1-
  from adsl;
  %let n_trt = &sqlobs;
quit;
Creates &trt1, &trt2, ... up to the number of rows. &sqlobs holds the count.

Clinical use: Building dynamic column lists for tables, looping through treatment groups or parameters, passing values between steps.
